0 · The pieces

Who's who, before the diagrams

Every box in the diagrams below is one of these six. Agents never talk to users, and the gateway never imports agent code — they only meet on the bus and in the Registry.

Gateway

The main FastAPI process (:8000) a user's prompt enters. It hosts the LangGraph state machine (Guard → Router → Planner → Orchestrator → Executor → Gate → Synthesizer) — the "brain" that plans work, dispatches it to agents, and assembles the final answer. Only the gateway talks to users.

src/genie/interface/bootstrap.py · application/graph.py
Agent service

A separate process per agent (:8010+), started by the run_agent harness. Exposes the agent two ways: synchronous JSON-RPC over HTTP, and a Kafka consumer on its own inbox topic genie.agents.{id}.inbox. Agents self-register with the Registry (:8002); the gateway discovers them there, so teams deploy agents without touching the platform.

src/genie/agents/server.py
interrupt()

LangGraph's "durable await". After producing a bus request, the Executor calls interrupt(): the entire run state is saved to a checkpoint and execution stops — the process could even restart. When the reply arrives, graph.invoke(Command(resume=results)) wakes the run exactly where it paused. On resume the node re-runs from the top, which is why correlation ids are deterministic: the re-produce is a duplicate that dedup drops.

application/nodes/executor.py · checkpointer.py
Dedup

Kafka delivers at least once — crashes, rebalances and the interrupt-resume re-produce all cause duplicates. Dedup turns that into an effectively-exactly-once system via atomic Redis claims: first claimant wins, everyone else drops the work. Used at three points: agent inbox (cid, attempt) so an agent never runs twice; per reply, so a real reply, a timeout and a cancel race to one winner; per group, so a fanned-out run resumes exactly once.

src/genie/messaging/dedup.py
ReplyRouter

The counterpart of interrupt() — a loop inside the gateway and the only consumer of the reply topic. For each record it reads the correlation_id header, looks up the matching a2a_awaiting record in Mongo, marks it resolved, and when the whole dispatch group is complete, resumes the suspended graph in a background task.

src/genie/interface/reply_router.py
Supervisor

The control plane (§ 2): a separate process that sweeps expired waits through the EXTEND → RETRY → DEAD-LETTER ladder and consumes the DLQ, unblocking stuck runs with step.cancelled control messages. It has no graph — all unblocking flows back through the bus.

src/genie/messaging/supervisor.py · services/supervisor/

How a reply finds its owner — nobody "recognizes" replies by reading them

Agents never read the reply topic, so agent C never has to decide "is this mine?". Matching is a database lookup keyed by an opaque ticket number — like a restaurant pager: the kitchen puts food on the counter with ticket #42; other diners aren't watching the counter at all.

1 · ticket written first Before producing, the Executor writes an a2a_awaiting record to Mongo: cid = uuid5(run_id:task_id:attempt), plus thread_id, run_id, group_id — who is waiting.
2 · agent echoes the ticket Agent B copies correlation_id from request headers into its reply headers unchanged. B never knows or cares who's waiting.
3 · router looks it up The ReplyRouter fetches the cid's record from Mongo — the record says which run this belongs to. Unknown cid → logged and ignored.
4 · any instance may win Multiple gateways share one consumer group, so each reply reaches exactly one instance — not necessarily the dispatcher. Fine: waiting state (Mongo), claims (Redis) and checkpoints are shared, so ownership is a property of the record, never the process.

Two implications: consumers of one reply topic must be one logical application sharing one awaiting store — that's why each team gets its own reply topic in § 4. And "not reading what isn't yours" is cooperative, not enforced: a modified agent could subscribe or forge a reply with an observed cid (§ 3) — ACLs are the enforcement layer.

note: agents never call each other over the bus — BaseAgent.call_peer is blocking HTTP JSON-RPC. Only the gateway's Executor dispatches via Kafka (an agent isn't a graph and can't suspend).

1 · What runs today

A LangGraph run suspends at the Executor, the agent works from its own inbox topic, and every reply — from every agent — lands on one shared topic that only the gateway's ReplyRouter consumes.

2 · Control plane

Supervisor: the failure ladder

A separate process (services/supervisor) sweeps expired a2a_awaiting records. It never touches the graph — all unblocking flows back through the bus.

deadline expired →

EXTENDagent heartbeat healthy and extensions left — slow, not dead. Push the deadline out.
RETRYattempts left — re-produce to the inbox with attempt+1. Deterministic cid keeps it deduped.
DEAD-LETTERexhausted — publish to genie.dlq with the error header.
DLQ consumer Supervisor permanently consumes genie.dlq and produces step.cancelled to the reply topic — the waiting run unblocks immediately and the Gate → Planner loop re-plans (fallback agent or partial answer).
Replay scripts/replay_dlq.py re-produces dead letters after a fix. With the Supervisor disabled, the gateway's simpler timeout sweep runs instead.
3 · The gap

Topic names are not a security boundary

Today every process connects with the same broker credential. The code never subscribes across agents — but Kafka doesn't enforce the code's conventions. Who can actually read what:

topicintended readeractual readability today
genie.agents.{a}.inbox agent a only any process with the bus credential
genie.replies (shared) gateway ReplyRouter any process with the bus credential
genie.dlq Supervisor any process with the bus credential
Read every reply. Subscribe genie.replies with a fresh consumer group — full Task output of every agent, every team. Consumer groups are namespaces, not permissions.
Read another agent's inbox. Often worse than replies: requests carry the args and the blackboard snapshot of the run.
Forge a reply. Observe a correlation_id on the wire, produce a fake Task to genie.replies — the ReplyRouter resolves it and resumes the victim's run with attacker data.

Conclusion: the fix is broker-enforced identity + ACLs, not topic layout. Renaming or splitting topics changes nothing while every client holds the same key.

4 · Multi-team target

Per-team namespaces, per-principal ACLs

Keep the shared-reply-topic pattern — but one per team, with the broker enforcing who reads and writes it. Invocation rights live in the Registry and are checked at dispatch.

Two layers, both required: Kafka ACLs control topic access; the Registry (AgentMeta gains team + allowed_callers) controls invocation — the gateway checks it before producing. The reply_to header already routes replies per request, so agents need no protocol change.

Code delta — smaller than it sounds

  1. messaging/envelope.py — make inbox_topic() / reply_topic() / dlq_topic() team-aware; actually populate HDR_TENANT_ID (the client already threads tenant_id through).
  2. platform/config.py — team identity per deployment; per-principal broker credentials in local.yaml / env, never default.yaml.
  3. interface/reply_router.py — subscribe genie.{team}.replies instead of the global topic.
  4. registry/agent_meta.py + gateway dispatch — team, allowed_callers, enforce at send_via_bus() time.
  5. Broker provisioning — an admin script (or Terraform) that creates principals + the ACL matrix above whenever an agent registers.
  6. Optional, only if broker admins are outside the trust domain: per-team envelope encryption of the Kafka value — headers stay plaintext so routing on headers keeps working.
5 · Cross-org

The bus never crosses the org boundary

Never hand another organization credentials to your cluster — shared blast radius, leaked metadata, operational coupling. The trust boundary lives at an HTTPS edge, where real authn/authz, audit and rate limiting exist. Your agents are already A2A services; that protocol is the cross-org transport.

org 1

internal Kafka bus team-a team-b namespaces + ACLs (§ 4) gateway · agents · supervisor

edge bridge: consumes local bus ↔ speaks A2A outward, feeds replies back to genie.{team}.replies

⇢  A2A over HTTPS  ⇠
EDGE, NOT BUS
agent card discovery

org 2

their own bus (or none) whatever they run internally — invisible to org 1 their agents expose A2A endpoints

async semantics? their bridge produces into their bus with credentials they issued themselves

Rule of thumb: Kafka inside a trust domain, A2A HTTPS between trust domains. A cross-org call looks exactly like a transport="json-rpc" agent to the Executor — no graph changes.
