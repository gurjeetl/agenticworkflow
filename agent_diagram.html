Base Agent Features

How every domain agent (Weather, Hotel) reuses one shared engine.

Base BaseAgent

The shared engine. Built once, used by every agent.

  • Talks to the LLM
  • Discovers & calls external tools (MCP)
  • Enforces user permissions on tools
  • Reads & writes shared memory & state
  • Tracks iterations & stops runaway loops
  • Emits traces & logs (MLflow)
  • Handles errors & final responses

Shared Conversation Context

How Weather → Hotel hand off data within one request.

  • Remembers: recent messages in this chat
  • Persists: user facts across sessions (MongoDB)
  • Tracks: which agent is up next
  • Guards: iteration count, completion, errors

Derived WeatherAgent

A specialist — same engine, its own persona & tool.

  • Persona: weather reporter
  • Calls one MCP tool (get_weather)
  • Formats the tool's reply for the user
  • Inherits everything else from BaseAgent

Derived HotelAgent

Another specialist — same engine, different persona & tool.

  • Persona: hotel concierge
  • Calls one MCP tool (get_hotels)
  • Formats the tool's reply for the user
  • Inherits everything else from BaseAgent
BaseAgent handles all of this:
  • LLM calls
  • MCP tool discovery & invocation
  • Permission checks on tools
  • Iteration & loop control
  • Error handling
  • Memory & state management
  • Logging & MLflow tracing
  • Final-response formatting

Each specialist on the right just adds its persona and tool →

Entire WeatherAgent file — ~12 lines

class WeatherAgent(BaseAgent):
    system_prompt = "You are a helpful weather reporter."
    tool_names = ["get_weather"]

    def run(self, state):
        city = (state.get("location") or "").lower().strip()
        return self.answer_with_tool(
            state,
            tool_name="get_weather",
            args={"city": city},
            format_text=lambda r: f"Weather for {city.title()}: {r}",
        )
